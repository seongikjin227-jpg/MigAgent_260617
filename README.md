# Migration Agent Pipeline

Version: 2026-06-17

Oracle/MyBatis 기반 DB migration, SQL conversion, SQL tuning, SQL formatting을 자동화하는 Python batch agent와 Streamlit 운영 콘솔입니다.

현재 코드는 다음 테이블을 중심으로 동작합니다.

- `NEXT_MIG_INFO`, `NEXT_MIG_INFO_DTL`: DB migration 및 기본 table/column mapping
- `NEXT_SQL_INFO`: XML에서 수집한 SQL 변환/튜닝 작업 상태
- `NEXT_SQL_LOG`: SQL 생성/실행/검증 stage별 상세 로그
- `NEXT_MIG_RAG_INFO`: SQL Conversion / SQL Tuning 통합 RAG rule
- `AG_AGENT_RUN_METRICS`: agent 실행 집계

## 주요 기능

- DB Migration: `NEXT_MIG_INFO` 기준 migration SQL 생성, 실행, 검증
- DB Migration dependency: `PRIOR_MAP_ID`와 같은 target table의 낮은 `PRIORITY` 선행 작업이 `PASS`일 때만 후속 작업 실행
- DB Migration retry: `ORA-00001` 중복키 실패가 발생하면 다음 LLM retry 프롬프트에서 `MERGE INTO` 전략을 강제
- SQL Conversion: MyBatis AS-IS SQL을 Oracle/MyBatis TO-BE SQL로 변환하고 source/target row count 검증
- SQL Tuning: 검증된 TO-BE SQL에 `NEXT_MIG_RAG_INFO` 기반 tuning RAG를 적용하고 tuned SQL 검증
- SQL Formatting: 최종 SQL을 `sql_indent_format_prompt.json` 기준으로 정렬해 `FORMATTED_SQL`에 저장
- Correct SQL stage bypass: `TOBE_CORRECT_SQL`, `BIND_CORRECT_SQL`, `TEST_CORRECT_SQL`이 있으면 해당 LLM 생성 stage를 건너뛰고 직접 사용
- Unified RAG Manager: SQL Conversion / SQL Tuning rule을 하나의 `NEXT_MIG_RAG_INFO` 테이블과 Streamlit 화면에서 관리
- Fail stage status: conversion/tuning 실패를 stage별 상태로 저장해 dashboard와 fail analysis에서 분포 확인
- LLM fallback: 모델 접근 오류 발생 시 `LLM_FALLBACK_MODELS` 순서로 재시도
- XML Export: `FORMATTED_SQL` 기준 MyBatis mapper XML 다운로드

## 실행 흐름

```text
python main.py
  -> Supervisor Agent
      -> DB Migration Agent
      -> SQL Conversion Agent
      -> SQL Tuning Agent
      -> SQL Formatting Agent
```

Supervisor는 cycle마다 각 agent의 pending job을 조회하고 batch size만큼 실행합니다. 종료 신호가 없으면 다음 cycle로 계속 진행합니다.

Agent별 실행 여부는 `.env` 또는 Streamlit sidebar에서 제어합니다.

```env
DB_MIGRATION_ONLY=false
SQL_CONVERSION_ONLY=true
SQL_TUNING_ONLY=false
SQL_FORMATTING_ONLY=false
```

- 모든 값이 `false`: 전체 agent 실행
- 하나 이상 `true`: `true`인 agent만 실행

## NEXT_SQL_INFO 상태 체계

### Conversion: `STATUS`

| 상태 | 의미 |
| --- | --- |
| `URGENT` | conversion 우선 처리 대기 |
| `READY` | conversion 일반 대기 |
| `PENDING` | conversion 대기 |
| `CONVERSION-PASS` | conversion 최종 성공 |
| `FAIL-TOBE` | TO-BE SQL 생성 또는 사용 stage 실패 |
| `FAIL-BIND` | bind SQL 생성/실행 또는 bind set 생성 stage 실패 |
| `FAIL-TEST` | test SQL 생성/실행 또는 검증 stage 실패 |
| `SKIP` | 사용자가 의도적으로 제외 |
| `NA` | mapping 미준비 등 처리 대상 제외 |
| `NULL` | conversion 대기 |

호환을 위해 기존 `PASS`, `FAIL`도 일부 조회/집계에서 읽습니다. 신규 저장은 `CONVERSION-PASS`, `FAIL-TOBE`, `FAIL-BIND`, `FAIL-TEST` 기준입니다.

### Tuning: `TUNED_TEST`

| 상태 | 의미 |
| --- | --- |
| `URGENT` | tuning 우선 처리 대기 |
| `READY` | tuning 일반 대기 |
| `TUNING-PASS` | tuning 최종 성공 |
| `PASS_NON_SELECT` | non-SELECT라 tuned validation을 생략했지만 성공 처리 |
| `FAIL-TUNED` | tuned SQL 생성 stage 실패 |
| `FAIL-BIND` | tuned test용 bind 관련 stage 실패 |
| `FAIL-TEST` | tuned test SQL 생성/실행 또는 검증 stage 실패 |
| `NULL` | tuning 대상 아님 또는 conversion 미완료 |
| `NA` | tuning 제외 |

호환을 위해 기존 `PASS`, `TUNING_PASS`, `FAIL`도 일부 조회/집계에서 읽습니다. 신규 저장은 `TUNING-PASS`, `FAIL-TUNED`, `FAIL-BIND`, `FAIL-TEST` 기준입니다.

## SQL Conversion 흐름

```text
NEXT_SQL_INFO polling
  -> SQL_LENGTH / MAP_TYPE 계산
  -> target mapping readiness 확인
  -> TO-BE SQL 생성 또는 TOBE_CORRECT_SQL 사용
  -> bind parameter 감지
  -> bind SQL 생성/실행 또는 BIND_CORRECT_SQL 사용
  -> BIND_SET 생성
  -> test SQL 생성/실행 또는 TEST_CORRECT_SQL 사용
  -> STATUS = CONVERSION-PASS 또는 FAIL-* 저장
  -> 성공 시 TUNED_TEST = READY
```

Conversion polling 대상:

- 포함: `URGENT`, `READY`, `PENDING`, 기존 `FAIL`, `FAIL-TOBE`, `FAIL-BIND`, `FAIL-TEST`, `NULL`
- 제외: `SKIP`, `NA`, `CONVERSION-PASS`, 기존 성공 `PASS`

`SqlInfoJob.source_sql`은 `EDIT_FR_SQL`이 있으면 그것을 우선 사용하고, 없으면 `FR_SQL_TEXT`를 사용합니다. SQL Conversion RAG 검색도 이 기준 SQL을 block 단위로 나누어 수행합니다.

## Correct SQL stage bypass

`NEXT_SQL_INFO`에 correct SQL 값이 있으면 해당 stage의 LLM 생성을 건너뜁니다.

| 컬럼 | 적용 stage | 동작 |
| --- | --- | --- |
| `TOBE_CORRECT_SQL` | TO-BE SQL 생성 | LLM 생성 대신 correct SQL을 TO-BE SQL로 사용 |
| `BIND_CORRECT_SQL` | Bind SQL 생성 | LLM 생성 대신 correct SQL을 사용하고 bind SQL 실행, bind set 생성은 계속 진행 |
| `TEST_CORRECT_SQL` | Test SQL 생성 | LLM 생성 대신 correct SQL을 사용하고 test SQL 실행/검증은 계속 진행 |

Correct SQL은 해당 stage의 생성만 대체합니다. 전체 PASS를 강제하지 않습니다.

## 통합 RAG 테이블

SQL Conversion과 SQL Tuning은 이제 하나의 `NEXT_MIG_RAG_INFO`를 사용합니다.

```sql
CREATE TABLE SFAADM.NEXT_MIG_RAG_INFO
(
    RAG_ID        NUMBER GENERATED BY DEFAULT AS IDENTITY,
    CATEGORY      VARCHAR2(100) NOT NULL,
    RULE_TYPE     VARCHAR2(30)  NOT NULL,
    SOURCE_TABLES VARCHAR2(1000),
    USE_YN        CHAR(1) DEFAULT 'Y' NOT NULL,
    GUIDANCE_TEXT CLOB,
    SOURCE_SQL    CLOB,
    TARGET_SQL    CLOB,
    HIT_CNT       NUMBER DEFAULT 0 NOT NULL,
    CREATED_AT    TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
    UPDATED_AT    TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT PK_NEXT_MIG_RAG_INFO PRIMARY KEY (RAG_ID),
    CONSTRAINT CK_NEXT_MIG_RAG_INFO_USE_YN CHECK (USE_YN IN ('Y', 'N')),
    CONSTRAINT CK_NEXT_MIG_RAG_INFO_CATEGORY CHECK (CATEGORY IN ('SQL_CONVERSION', 'SQL_TUNING')),
    CONSTRAINT CK_NEXT_MIG_RAG_INFO_RULE_TYPE CHECK (RULE_TYPE IN ('GENERAL', 'SEARCH'))
);
```

권장 인덱스:

```sql
CREATE INDEX SFAADM.IX_NEXT_MIG_RAG_INFO_01
ON SFAADM.NEXT_MIG_RAG_INFO (CATEGORY, RULE_TYPE, USE_YN);
```

컬럼 의미:

| 컬럼 | 의미 |
| --- | --- |
| `CATEGORY` | `SQL_CONVERSION` 또는 `SQL_TUNING` |
| `RULE_TYPE` | `GENERAL` 또는 `SEARCH` |
| `SOURCE_TABLES` | comma-separated source table list. 예: `ASIS_CODE,ASIS_USER`. 비어 있으면 전체 공통 rule |
| `GUIDANCE_TEXT` | LLM에 전달할 가이드 문구 |
| `SOURCE_SQL` | conversion은 FROM SQL 예시, tuning은 bad SQL 예시 |
| `TARGET_SQL` | conversion은 TOBE SQL 예시, tuning은 tuned SQL 예시 |
| `HIT_CNT` | 실제 프롬프트에 들어간 SEARCH rule hit count |

### RAG 조회 규칙

공통:

- `USE_YN='Y'`만 사용
- `SOURCE_TABLES`가 비어 있으면 모든 job에 매칭
- `SOURCE_TABLES`가 있으면 `NEXT_SQL_INFO.TARGET_TABLE`에 저장된 table list와 token 단위로 매칭
- SQL 본문에서 table을 파싱해 RAG 필터에 보태지 않습니다. `SOURCE_TABLES`는 운영자가 직접 입력합니다.

`GENERAL`:

- vector search를 하지 않습니다.
- `CATEGORY`, `USE_YN`, `SOURCE_TABLES` 조건이 맞으면 mandatory guidance로 항상 프롬프트에 포함합니다.

`SEARCH`:

- 현재 SQL을 main SQL / subquery block으로 나누고 block마다 vector search를 수행합니다.
- embedding 사용 가능 시 FAISS vector search를 사용합니다.
- embedding 환경이 없거나 실패하면 token fallback search를 사용합니다.
- 실제 프롬프트에 포함된 `SEARCH` rule만 `HIT_CNT`를 증가시킵니다.

### SQL Conversion RAG

- `CATEGORY='SQL_CONVERSION'`
- `RULE_TYPE='GENERAL'`: conversion mandatory guidance
- `RULE_TYPE='SEARCH'`: 현재 `source_sql` 또는 subquery block과 `SOURCE_SQL` 예시를 비교해 top-k 검색
- prompt section:
  - `[SQL_CONVERSION_GENERAL_RAG_GUIDANCE]`
  - `[SQL_CONVERSION_SEARCH_RAG_TOP_K_BY_SQL_BLOCK]`
- SEARCH prompt item은 `matched_block_sql`, `guidance`, `source_sql`, `target_sql` 구조로 전달됩니다.

### SQL Tuning RAG

- `CATEGORY='SQL_TUNING'`
- `RULE_TYPE='GENERAL'`: tuning mandatory guidance
- `RULE_TYPE='SEARCH'`: 현재 TO-BE SQL 또는 subquery block과 `SOURCE_SQL` 예시를 비교해 top-k 검색
- prompt에는 `source_sql/example_bad_sql`, `target_sql/example_tuned_sql` 기준으로 전달됩니다.

## SQL Tuning 흐름

Tuning polling 대상:

- 조건: `STATUS IN ('CONVERSION-PASS', 'PASS')`, `TO_SQL_TEXT IS NOT NULL`
- 포함: `TUNED_TEST IN ('URGENT', 'READY', 기존 'FAIL', 'FAIL-TUNED', 'FAIL-BIND', 'FAIL-TEST')`

SELECT:

```text
TO-BE SQL
  -> SQL_TUNING RAG 조회
  -> TUNED_SQL / TUNED_RESULT 생성
  -> baseline TO-BE vs tuned SQL 비교 test SQL 생성/실행
  -> TUNED_TEST = TUNING-PASS 또는 FAIL-TEST
  -> 성공 시 FORMATTED_SQL 생성
```

INSERT/UPDATE/DELETE:

```text
TO-BE SQL
  -> SQL_TUNING RAG 조회
  -> TUNED_SQL / TUNED_RESULT 생성
  -> tuned validation 생략
  -> TUNED_TEST = PASS_NON_SELECT
  -> FORMATTED_SQL 생성
```

별도 SQL Formatting agent는 보정용입니다. `TUNED_TEST`가 tuning pass 계열인데 `FORMATTED_SQL`이 비어 있는 row만 다시 포맷팅합니다.

## 운영 메모

- README와 prompt JSON은 UTF-8 저장을 권장합니다.
- `NEXT_SQL_INFO.LOG`는 row의 최신 요약이고, 상세 stage 이력은 `NEXT_SQL_LOG`에 append-only로 저장됩니다.
- `FORMATTED_SQL`은 XML export 기준 SQL입니다.
- `NA`는 mapping 미준비 등 처리 대상 제외 상태입니다.
- `SKIP`은 사용자가 의도적으로 제외한 상태입니다.
- fail analysis는 conversion `STATUS`의 `FAIL-*`, tuning `TUNED_TEST`의 `FAIL-*`를 기준으로 stage 분포를 보여줍니다.
